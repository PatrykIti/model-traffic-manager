## Community support

Use GitHub issues for:

- bug reports
- feature requests
- documentation problems
- operator questions that are safe to discuss in public

When opening an issue, include:

- the router config shape involved
- the exact command or request path
- the observed behavior
- relevant logs with secrets removed

## Security issues

For vulnerabilities or anything that could expose credentials, tenants, or traffic incorrectly, follow [SECURITY.md](./SECURITY.md) instead of opening a public issue.

## Sponsorship

If this project is useful to your team, the best way to support ongoing work is through the repository sponsor button or [GitHub Sponsors](https://github.com/sponsors/PatrykIti).

Sponsorship helps fund:

- documentation and examples
- reliability and live validation work
- release hardening
- roadmap time for new features

## Production adoption and design-partner interest

If you are evaluating `model-traffic-manager` for real production use, open an issue describing your use case and constraints. That is currently the best path for:

- deployment questions
- roadmap feedback
- design-partner conversations
- supportability gaps that block adoption
